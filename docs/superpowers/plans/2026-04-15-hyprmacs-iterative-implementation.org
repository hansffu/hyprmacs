* Goal
Build a Hyprland plugin plus Emacs package that makes one Hyprland workspace behave like an Emacs-directed workspace without replacing Hyprland's compositor, popup, or input responsibilities.

* Architecture Summary
Hyprland remains the compositor and input authority. The plugin owns workspace policy for a managed subset of tiled clients and exposes a private Unix socket IPC interface. Emacs owns buffer association, visible-set selection, and rectangle computation, then publishes snapshot-style layout updates to the plugin.

* Default V1 Decisions
- One Emacs controller process and one managed workspace per active session.
- Plugin implementation in C++.
- Emacs package implementation in Emacs Lisp.
- The existing flake-based nested Hyprland environment is the base developer workflow and should be extended rather than replaced.
- IPC transport is a plugin-owned Unix domain socket under =$XDG_RUNTIME_DIR=.
- IPC payloads are newline-delimited JSON with an explicit protocol version field.
- Overlapping managed rectangles are rejected in v1 instead of guessed.
- No resize animations in v1.
- If Emacs disconnects, the plugin unmanages the workspace and returns clients to native Hyprland behavior.
- Floating windows, dialogs, popups, child frames, and posframes remain native Hyprland surfaces.
